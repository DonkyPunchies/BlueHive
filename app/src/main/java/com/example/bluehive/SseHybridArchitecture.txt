  CHANGELOG — READ THIS FIRST IF YOU LAST READ THE 07-13 VERSION
────────────────────────────────────────────────────────────────────────────────

The 07-13 revision of this document described all four TV→server session calls
as live and the whole system as living on the PLATFORM backend. Both statements
are now false. Two refactors happened:

  PHASE 2 — the TV stopped making three of its four session calls.
      background(), foreground() and endSession() were DELETED from the Android
      client. They targeted the platform identity backend, which had stopped
      accepting BlueHive's host-injected token. Slot lifecycle was handed to the
      SSE connect/disconnect lifecycle instead.

  PHASE 3 (2026-07-21) — session state moved to bluehive-api.
      is_device_active / backgrounded_at / last_seen_at / slot now live in the
      bluehive.device_sessions table owned by bluehive-api. The TV's SSE stream
      and its remaining check-slot call were repointed off platform.

  !! THE BUG THAT SAT BETWEEN THEM — do not re-introduce it !!
      PHASE 2 delegated everything to the SSE lifecycle. But BlueHive's SSE had
      been silently failing against platform since the host/companion model
      landed: platform resolves an event stream by (X-Device-Fingerprint,
      user_id) against platform_accounts.devices, and in the companion model
      that row belongs to the HOST (Off-Grid Drive) — while
      SessionManager.deviceFingerprint is a SHA-256 of BLUEHIVE'S OWN
      ANDROID_ID. The two never matched, so every connect got
      404 "Device not found" and retried forever (2s, 4s, 8s, 16s...).

      Net effect: the pings were gone AND the stream never opened, so NOTHING
      set or cleared is_device_active. That is the "is_device_active is stuck"
      bug. PHASE 3 fixed it — bluehive-api identifies the session from the
      `device_id` CLAIM IN THE JWT, so no fingerprint match is required.

      Lesson: fingerprint-based device resolution is wrong for a companion app.
      The host owns the device row; the companion only ever holds a token.

  PHASE 3 platform-side — DONE 2026-07-21/22:
      Platform's /api/android/events is now a pure revocation relay (no
      is_device_active writes, no cap), the cap at pairing/refresh was removed,
      and is_device_active / backgrounded_at were DROPPED from
      platform_accounts.devices entirely (D4). Platform owns identity only; the
      6-device cap is bluehive-api's. See §11.

  PHASE 3.1 — DEVICE-LIFECYCLE ACCURACY RESTORED (2026-07-22). This is the big
  one for anyone who read a prior version and thinks the pings are gone:
      The 3 explicit lifecycle pings that PHASE 2 removed are BACK, now pointed
      at bluehive-api (which accepts BlueHive's token — the reason PHASE 2
      removed them was that they hit PLATFORM, which stopped accepting it):
          Back button  → POST /end-session  → inactive INSTANTLY
          Home button  → POST /background   → 15-min grace key
          return       → POST /foreground   → cancel the grace
      Why restored: the SSE-disconnect-only model could not tell a Back (instant)
      from a Home (grace) — both just drop the connection — and its finally-block
      teardown was silently failing (see below). The explicit pings are precise.

      NEW: a stale_session_reaper (bluehive-api lifespan, every 120s) flips
      is_device_active=False for any session whose keepalives stopped >15 min ago
      — the crash / power-cut / internet-drop cases that send no ping. This is
      the ONLY thing that corrects the RAW column for no-signal closes.

      BUG FIXED AT THE SOURCE: the SSE finally's is_device_active DB write was
      REMOVED. On a client disconnect uvicorn cancels the request task, so that
      write's `await commit()` died with CancelledError and the flip never landed
      ("is_device_active stuck true after a proper close"). The reaper + pings own
      the state now, so the finally only tears down the pubsub — no DB, no race.

  1. WHAT IS THIS SYSTEM?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BlueHive's backend needs to instantly communicate with Android TV devices the
moment something important happens — a device is removed, a session is revoked,
or a slot opens up in a full workspace. Rather than having every TV constantly
ask the server "has anything changed?", the server tells the TV the moment it
does.

This is done using SSE (Server-Sent Events) — a permanent, one-way connection
from the server to each TV that stays open and silent until something actually
happens. When it does, the event arrives in under 50 milliseconds.

  Connection direction ......... Server → TV (server pushes, TV listens)
  Connection type .............. One long-lived HTTP connection per TV (SSE)
  Served by .................... bluehive-api  (NOT platform-api — see §3)
  TV → Server communication .... Four short REST calls, all EVENT-driven (never
                                  polled): end-session (Back), background (Home),
                                  foreground (return), check-slot (lockout retry).
                                  Restored in PHASE 3.1 — see the changelog.
  Idle network usage ........... Zero — connection is silent when nothing
                                  is happening
  Event delivery latency ....... Under 50ms on local network
  Max active devices ........... 6 per workspace, enforced by bluehive-api
                                  at SSE connect and at check-slot

(This section is unchanged — the rationale still holds.)

THE OLD WAY (POLLING) — WHAT WAS REPLACED
------------------------------------------
The old HeartbeatManager sent a POST request every few seconds to ask "am I
still okay?" This meant constant network traffic even when nothing was
happening, up to N seconds of delay on revocations, and server load that grew
linearly with the number of active devices.

WHY NOT WEBSOCKET?
------------------
WebSocket is full-duplex (both sides can talk at any time) but everything
meaningful in this system only goes one direction: the server tells the TV
things. SSE is a plain HTTP response, works through proxies and load balancers
without configuration, and is simpler to implement.

WHY SSE?
--------
  • Zero bytes sent during idle — no polling noise
  • Server pushes exactly when something changes
  • Under 50ms delivery on LAN
  • Plain HTTP — works through any proxy or firewall
  • Automatic reconnect built into the protocol
  • Server re-checks device status on every reconnect — missed events are
    caught automatically

  3. WHICH BACKEND OWNS WHAT  (NEW — this is the PHASE 3 split)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BlueHive and Off-Grid Drive are separate products sharing one identity platform.
Two backends, two databases, and they never query each other's tables.

  PLATFORM-API  (platform-backend, platform_accounts schema, :9000)
      Owns IDENTITY. Accounts, workspaces, device pairing, refresh tokens,
      installed add-ons. Issues the access JWT.
      BlueHive still calls it for exactly two things: /api/android/refresh
      (indirectly, via the host) and /api/me.

  BLUEHIVE-API  (bluehive-backend, bluehive schema, :8000)
      Owns STREAMING SESSIONS. The SSE stream, the 6-device cap, and the
      bluehive.device_sessions table. Also profiles, watch history, favourites.

  THE ONLY COUPLING IS THE JWT.
      bluehive-api verifies platform-signed HS256 tokens IN-PROCESS with a
      shared JWT_SECRET_KEY. It never calls platform-api and never touches
      platform_accounts. Claims it consumes:

          sub        → user id
          device_id  → WHICH DEVICE. This is the session key. Opaque integer.
          ws         → workspace id, so the cap can be scoped without a lookup
          email      → informational

      Because device_id comes from the token, bluehive-api needs no device row
      of its own to resolve a stream. This is precisely what fixed the 404 bug.

  REDIS: each backend has its OWN.
      bluehive-redis (in bluehive's docker-compose, --notify-keyspace-events Ex)
      carries BlueHive's channels and grace keys. Platform's redis is separate.
      There is deliberately NO platform→bluehive Redis link — that would be
      direct coupling between two products that are meant to look independent.

  REVOCATION DOES NOT TRAVEL OVER THIS SSE.
      A kick is an IDENTITY event, so it flows through the host:
        platform → OGD's own SSE → AIDL onIdentityRevoked() → BlueHive.
      Measured end-to-end at 63ms. BlueHive's DeviceEventStream still has
      "revoked"/"unpaired" handlers, but bluehive-api does not publish those —
      they are a dormant legacy path, kept because they cost nothing.

  4. THE TWO REDIS CHANNELS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Both channels now live on bluehive-redis.

  bh:device:{fingerprint}
      Private channel for one specific TV. Used for force_close.

  bh:workspace:{user_id}
      Broadcast channel for all TVs on the same account. Used for slot_freed
      events so every TV on the lockout screen can react simultaneously.

  bh:bg:{fingerprint}
      Not a channel — the 15-minute background grace KEY, set by POST /background
      (Home button). Its expiry (via keyspace notifications) is what frees a
      backgrounded device's slot if the user never returns.

  NOTE ON {fingerprint}: this is whatever the client sent in
  X-Device-Fingerprint, and for BlueHive that is BlueHive's own ANDROID_ID hash
  — NOT the host's device fingerprint stored in platform_accounts.devices. That
  is fine: the value is only ever used to route Redis keys back to the same
  session, and bluehive-api writes it and reads it consistently. The DATABASE
  identity is device_id. Never join these two backends on fingerprint.

  5. DEVICE STATE — HOW A TV OCCUPIES A SLOT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All of these are columns on bluehive.device_sessions (NOT on
platform_accounts.devices, which keeps only pairing identity).

  row exists, keyed by device_id
      Created on first contact (SSE connect or check-slot). device_id is UNIQUE
      and carries no foreign key — the two databases are never joined.

  is_device_active = True
      The TV is currently in the app with an open SSE stream.

  is_device_active = False
      The TV is paired but not actively in the app.

  last_seen_at
      Refreshed by the SSE keepalive. THIS IS NOT MERELY AN AUDIT FIELD — read
      the box below. It is half of the real definition of "active".

  backgrounded_at
      Set by POST /background (Home button). Server-side only; the 15-minute
      grace timer runs as a Redis key with a TTL. Cleared by /foreground.

  is_device_active is flipped FALSE by exactly three things (PHASE 3.1):
      • POST /end-session        — Back button, instant
      • grace-expiry listener    — Home, 15 min, no return
      • stale_session_reaper     — crash / net drop, 15 min after last keepalive
  It is set TRUE only by SSE connect and check-slot. The SSE finally NO LONGER
  writes it (that write raced request-cancellation — see §10). During a grace
  window it deliberately STAYS true: a backgrounded device still holds its slot.

  ┌──────────────────────────────────────────────────────────────────────────┐
  │  WHAT "ACTIVE" ACTUALLY MEANS                                            │
  │                                                                          │
  │  A device counts against the cap only when BOTH are true:                │
  │                                                                          │
  │      is_device_active = True   AND   last_seen_at >= now - 15 minutes    │
  │                                                                          │
  │  The raw boolean is now kept honest by the reaper (it flips stranded     │
  │  rows within a couple minutes of the 15-min line), but reporting still   │
  │  applies BOTH conditions as defence in depth. check_workspace_active_    │
  │  count() and /api/bluehive/device-states both do this — the latter is    │
  │  why a TV that died is no longer shown "Active" to the web dashboard.    │

  6. EVENTS — WHAT GETS PUSHED AND WHEN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

EVENTS BLUEHIVE-API PUSHES TO THE TV
--------------------------------------
  slot_freed
      A workspace slot just opened up. TVs sitting on the "workspace full"
      lockout screen use this to show a green "A slot just opened" hint.

  force_close
      Internal signal telling the SSE generator to close cleanly. Sent by
      end-session so teardown does not depend on TCP disconnect detection.

  session_started
      Broadcast when a TV connects, so the web devices panel can refresh.

  revoked / unpaired  — NOT PUSHED BY BLUEHIVE-API.
      The client still parses them, but kicks arrive over the host AIDL
      callback described in §3. Kept as a dormant fallback only.

CALLS THE TV MAKES TO THE SERVER   (all LIVE again as of PHASE 3.1)
----------------------------------
  All four ride ApiClient.bluehiveApi, whose interceptor attaches the host
  Bearer, X-API-Key, X-BlueHive-Version AND X-Device-Fingerprint. That last
  header matters: bluehive-api resolves the grace key by fingerprint, so a
  missing one would silently stop the grace from ever freeing the slot.

  POST /api/android/end-session   — Back button (proper close). Flips
      is_device_active=False immediately + pushes force_close/slot_freed.
      Caller: CloseApplication.performClose (2s timeout, then finishAffinity).

  POST /api/android/background    — Home button. Sets the 15-min grace key.
      Caller: InactivityWatcher.onAppBackgrounded (skipped if a Back-close
      already ran, guarded by DeviceEventStream.isIntentionallyExited).

  POST /api/android/foreground    — return from background. Deletes the grace
      key. Re-activation + cap check are the SSE reconnect's job, not this.
      Caller: InactivityWatcher.onAppForegrounded.

  POST /api/android/check-slot    — Refresh on the lockout screen after the
      countdown. Runs the capacity check and re-occupies a slot if free.
      Caller: LockoutActivity.

  If any ping is lost (network blip as the app closes), the stale_session_reaper
  is the backstop — the device still flips inactive ~15 min after its last
  keepalive. So the pings make it PRECISE; the reaper makes it CERTAIN.

  7. HOW EACH SCENARIO WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

──────────────────────────────────────────
  NORMAL APP LAUNCH
──────────────────────────────────────────
  [1] App opens
      BlueHiveApplication.onCreate() runs. SessionManager loads stored
      credentials. In the host model the access token is injected by the host
      over AIDL rather than being fetched by BlueHive itself.

  [2] SSE stream opens
      DeviceEventStream.start() fires GET /api/android/events at BLUEHIVE-API.
      Server verifies the JWT, requires it to be a TV token, and reads
      device_id + ws from the claims.

  [3] Cap is enforced BEFORE the slot is taken
      check_workspace_active_count() counts other sessions in the workspace
      that are active AND seen within 15 minutes. At the limit → 429. The
      calling device excludes itself, so re-entry never counts against its
      own slot.

  [4] Session row upserted, Redis subscribed
      is_device_active = True, last_seen_at stamped, any pending grace key
      deleted. Server subscribes to the device and workspace channels.

  [5] Stream sits silent
      No bytes exchanged until an event happens. A keepalive comment every
      30s keeps the connection alive through proxies and refreshes
      last_seen_at. The keepalive deliberately does NOT write is_device_active.

──────────────────────────────────────────
  HARD REMOVE (Admin Deletes the Device)
──────────────────────────────────────────
  Handled entirely on the IDENTITY path, not this SSE:
  platform deletes the device row → pushes to the OGD host's stream → OGD
  broadcasts onIdentityRevoked() over AIDL → BlueHive clears its session and
  routes to the pairing screen. No countdown.

──────────────────────────────────────────
  SOFT KICK (Admin Revokes a Session)
──────────────────────────────────────────
  Same identity path as above; the host distinguishes the two. A soft kick
  leaves the pairing intact and BlueHive shows the lockout countdown, then
  re-enters via check-slot.

  !! The host MUST NOT treat "revoked" as terminal. Conflating it with
     "unpaired" wipes the TV's pairing on what should be a recoverable kick.

──────────────────────────────────────────
  HOME BUTTON — 15-MINUTE GRACE TIMER
──────────────────────────────────────────
  [1] User presses the home button
      ProcessLifecycleOwner.onStop() fires. InactivityWatcher.onAppBackgrounded
      POSTs /api/android/background → bluehive-api sets bh:bg:{fp} EX 900. The
      device STAYS is_device_active=True during grace (it still holds its slot).

  [2] User returns within 15 min
      InactivityWatcher.onAppForegrounded POSTs /api/android/foreground → grace
      key deleted. If the SSE had dropped, startAfterUserReturn() also reconnects
      (connect clears the key + re-activates through the cap). Slot never freed.

  [3] User does NOT return
      The bh:bg key expires at 15 min → the keyspace listener flips
      is_device_active=False + pushes slot_freed. (If the SSE also died and
      last_seen went stale, the reaper would reach the same result — belt and
      suspenders.)

  The KNOWN GAP from the SSE-only era (SSE survives a brief background, no
  foreground signal) is CLOSED: /foreground is now sent explicitly on return.

──────────────────────────────────────────
  WORKSPACE FULL — SLOT FREED PATH
──────────────────────────────────────────
  [1] A 7th TV tries to connect
      GET /api/android/events returns 429. LockoutBus posts
      REASON_WORKSPACE_FULL. DeviceEventStream keeps retrying.

  [2] Another TV frees its slot
      Clean exit, grace expiry, or simply ageing out of the 15-minute window.
      slot_freed is pushed to the workspace channel.

  [3] Waiting TV receives slot_freed
      SlotFreedBus fires; LockoutActivity shows the green re-entry hint.

  [4] User taps Refresh
      POST /api/android/check-slot re-runs the capacity check.
      200 = enter. Still full = 429.

BLUEHIVE-API  (bluehive-backend — owns sessions and slots)
-----------------------------------------------------------

  bluehive_plugin/device_sessions.py
      The heart of the system now. Contains:
        GET  /api/android/events           — the persistent TV stream
        POST /api/android/check-slot       — re-entry after lockout
        POST /api/android/end-session      — Back button (instant inactive)
        POST /api/android/background       — Home button (15-min grace)
        POST /api/android/foreground       — return (cancel grace)
        GET  /api/bluehive/device-states   — read-only, for platform to merge
                                             into the web devices panel
        check_workspace_active_count()     — the single cap enforcement point
        bg_expiry_listener()               — reacts to grace-key expiry
        stale_session_reaper()             — 120s sweep; flips is_device_active
                                             False for sessions stale >15 min
                                             (the no-signal crash/disconnect case)

  bluehive_plugin/bh_models.py
      DeviceSession ORM model. device_id is UNIQUE and has NO foreign key —
      the platform database is a different database entirely.

  bluehive_plugin/bh_auth.py
      Verifies the platform-signed JWT in-process and exposes TokenIdentity
      (user_id, device_id, workspace_id).

  bluehive_plugin/bh_redis.py
      Shared bluehive-redis connection.

  TMDB_Api_Server.py
      Mounts the router and starts bg_expiry_listener() in the lifespan.

  9. STRENGTHS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

RELIABILITY
-----------
  • Every SSE reconnect is also a status check — missed events during network
    outages are caught the moment connectivity returns, without polling.

  • The 15-minute grace timer runs entirely in Redis server-side. Android's
    Cached App Freezer cannot affect it — it fires with exact wall-clock
    precision regardless of what happens to the TV process.

  • The 15-minute last_seen_at window handles every unhandled exit path —
    crashes, force-stops, power loss — without any special code. It is not a
    nicety; it is what makes the coarse SSE-lifecycle approach safe.

  • Session identity comes from the JWT, so a companion app with no device row
    of its own can still be resolved. This is what makes the host model work.

CORRECTNESS
-----------
  • is_device_active transitions (PHASE 3.1):
        Set True:   SSE connect, check-slot
        Set False:  /end-session (Back), grace-key expiry (Home), reaper (crash)
    The keepalive deliberately does not write it, preventing race-resurrection
    of ended sessions. The SSE finally does NOT write it either (that raced
    request-cancellation — §10). The reaper guarantees the False paths converge,
    but reporting still pairs the boolean with the freshness window as defence.

  • Hard remove and soft kick are distinct paths with distinct events and
    distinct client behaviour. They must not be conflated.

  • check_workspace_active_count is the single enforcement point for the cap,
    and it excludes the calling device so re-entry cannot lock a user out of
    their own slot.

  • The SSE client has no TokenAuthenticator — it rotates its token before
    connecting, so a mid-stream 401 cannot silently kill the connection.

SIMPLICITY
----------
  • Zero polling anywhere, including between the two backends. The web panel's
    read of device-states is on-demand, not a heartbeat.

  • Redis channels are the only coupling between the HTTP layer and the SSE
    layer. New event types need no new infrastructure.

  10. WEAK POINTS & KNOWN RISKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

THE SSE `finally` NO LONGER WRITES THE DB     ← was a bug; fixed PHASE 3.1
-------------------------------------------------------
  History (the bug):
    Teardown used to set is_device_active=False in the generator's finally. TWO
    ways it failed: (a) a hard client kill (force-stop/power-cut) leaves the
    generator suspended at its yield, so Starlette never closes it and finally
    never runs; (b) even on a clean disconnect, uvicorn CANCELS the request task,
    so the finally's `await commit()` died with CancelledError ("Exception
    terminating connection <asyncpg>") — the flip never landed. Net symptom:
    is_device_active stuck True after a proper close, no "marked inactive" log.

  Fix (current):
    The finally's DB write was REMOVED — it only tears down the pubsub now (no
    DB, no cancellation race). is_device_active is instead owned by the explicit
    pings (Back=/end-session, Home=/background→grace listener) and the
    stale_session_reaper (crash/disconnect, 15-min sweep). Do NOT re-add a DB
    write to the finally — it will hit the same cancellation and the reaper
    already covers the no-signal case.

SINGLE REDIS INSTANCE
----------------------
  Risk:
    If bluehive-redis goes down, the push helpers catch the exception and log a
    warning — the HTTP response still succeeds and the DB commit still happens,
    but the TV does not get the event in real time.

  Mitigation:
    The reconnect status check preserves correctness for anything that matters;
    slot_freed degrades to a manual Refresh. Sentinel/Cluster at scale.

NO PERSISTENT EVENT LOG
------------------------
  Risk:
    Redis Pub/Sub is fire-and-forget. An event published while no stream is
    subscribed is gone. slot_freed has no recovery path.

  Mitigation:
    Redis Streams (XADD/XREAD with consumer groups) if this ever matters.

SINGLE BACKGROUND LISTENER TASK
---------------------------------
  Risk:
    bg_expiry_listener() runs as one asyncio task per process. With multiple
    uvicorn workers, every worker receives each keyspace notification and they
    all write. Single worker today, so dormant.

  Mitigation:
    A Redis SET NX EX lock inside the listener. Required before scaling out.

HARDCODED LOCAL IP (Development Only)
---------------------------------------
  Risk:
    The base URLs are LAN addresses over plain HTTP, so bearer tokens transit
    in cleartext. Intentional for development, a blocker for production.
    Note ApiClient hardcodes its two base URLs as constants while
    DeviceEventStream reads BuildConfig — those are two sources of truth for
    the same host and they must be changed together.

  Mitigation:
    Point the release build config at production HTTPS URLs before any public
    deployment, and consider collapsing the duplicated base-URL constants.

  12. EDGE CASES — QUESTIONS & ANSWERS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: User switches apps temporarily (home button, notifications)?
A: App backgrounds. No HTTP call is made any more. If the process survives and
   the SSE stays open, nothing changes. If the process is killed, the stream
   dies and the slot frees via teardown or the 15-minute window.

Q: User swipes the app away (recent apps force-close)?
A: Process killed, SSE dies, last_seen_at stops updating. The generator's
   finally probably never runs (see §10), so the boolean may stay True — but
   the slot frees after 15 minutes (reaper + freshness window) with no clean exit.

Q: Why is is_device_active True for a TV that is obviously off?
A: Expected, and harmless. Read the box in §5. If you are looking at the WEB
   dashboard and it says "Active" for a dead TV, that IS a bug — device-states
   is supposed to apply the freshness window before reporting.

Q: Redis is down when something is pushed?
A: The push helper catches and logs. The DB commit already happened. The TV
   picks up state on the next reconnect. Correctness preserved, latency higher.

Q: Access token expires while the SSE stream is open?
A: SSE reads the response body — it does not re-authenticate mid-stream. Token
   expiry does not interrupt it. The next connect rotates the token first.

Q: BlueHive gets a 404 on connect. What broke?
A: Almost certainly that something repointed the stream back at platform-api.
   bluehive-api does not 404 on identity — it creates the session row from the
   JWT. Check the URL before anything else. See the changelog at the top.

Q: GeckoView is playing a movie when a lockout fires?
A: MainWebViewer subscribes to LockoutBus in onCreate. On any lockout event,
   initiateGracefulShutdown() closes the GeckoSession and releases media
   resources before the lockout screen appears.

Q: Multiple /background POSTs from OkHttp retries?
A: The endpoint checks whether a grace key already exists (TTL > 0). If so the
   call is classified "retry" and the EXISTING TTL is preserved — the timer is
   not reset. Verified: a repeat call showed TTL 599, not 600.

Q: 6 devices active, a kicked device presses Refresh as another exits?
A: check_workspace_active_count runs a counting query before marking anything
   active. The call that arrives while the count is at 5 succeeds; simultaneous
   ones still see 6 and get 429.
